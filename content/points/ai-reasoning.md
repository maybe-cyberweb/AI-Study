# Reasoning 与 Test-Time Compute：把“会想”变成“能算得出”

## 你真正要解决的问题
在生产系统里，“看起来很聪明”不等于“可交付”。推理型能力要能做到：
- 可控：失败时有边界、有降级。
- 可评测：能用数据集回归，不靠感觉。
- 可预算：推理时间/Token/工具调用次数能算清楚。

## 1) Test-Time Compute 是什么
训练时算力（train-time compute）决定模型的“上限”；  
推理时算力（test-time compute）决定你能否把上限“拿出来”。

常见做法：
- 多采样 + 选择：生成多个候选答案，挑最一致/最可验证的。
- 规划与搜索：先分解任务，再逐步执行与验证。
- 反思与自检：引入检查器（规则/工具/判别器）对输出进行验收。

## 2) 为什么工程上要做“推理预算”
推理越强往往意味着：
- 更长的思考（更多 token）→ 更贵、更慢。
- 更频繁的工具调用 → 更复杂的失败模式（超时、死循环、数据错误）。

因此必须把“推理”放进 Harness 的预算体系：
- 单次对话最大 token
- 单次对话最大工具调用次数
- 单次对话最大重试次数

## 3) 让推理可控的关键：把正确性绑定到“可验证”
在你们项目里，最有效的方式不是让模型一直想，而是：
- 让模型输出 **结构化计划**（步骤、参数、预期结果）
- Harness 逐步执行并验收（传感器/校验器）
- 失败就回传错误，触发改写计划

## 4) 面向干部教育场景的“推理增强”建议
- 政策问答：把“引用段落”作为验收条件（无引用不出结果）。
- 推课：把“课程 ID 存在 + 学习路径符合规则”作为验收条件。
- 测评：把“评分依据 + 证据项”作为验收条件，避免拍脑袋。

## 自检清单
- 我能写出推理预算策略（token/tool/retry）并解释阈值为何合理？
- 我能定义一个“可验收输出”（必须包含哪些字段/引用/证据）？
- 我能用回归测试集证明更强推理没有带来退化/幻觉上升？
