# KV Cache：决定推理吞吐与成本的关键

## 先把推理拆成两段
LLM 推理通常分为：

- Prefill：把输入 prompt 全部跑一遍，得到每层的 K/V
- Decoding：逐 token 生成，每次只算新 token 的 Q，并复用历史 K/V

KV Cache 就是“复用历史 K/V”的缓存。

## 1) 为什么 KV Cache 能加速
没有 cache：每次生成一个 token 都要把整段序列重新算 attention。

有 cache：历史 token 的 K/V 直接拿来用。

收益：

- 生成阶段从 O(T^2) 变成更接近 O(T)

## 2) 代价：显存
cache 保存的是每层每个 token 的 K/V。

因此：

- 上下文越长，显存线性增长
- batch 越大，显存线性增长

你会遇到的真实问题：

- 长对话会把显存吃满
- 并发时吞吐受限

## 3) 工程策略
- 滑动窗口：只保留最近 N 个 token 的 KV
- 摘要压缩：中间历史用摘要替换
- 批处理：把多个请求合并，提高吞吐

## 与项目落地的连接
干部伴读是典型“长对话+高并发”：

- 必须有会话摘要与锚点提示
- 必须有成本预算与并发控制
