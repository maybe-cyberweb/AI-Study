# Prompt 工程：可控输出的结构化写法

## 目标
把模型从“自由发挥”切换到“按规则产出”。你真正需要的是：

- 约束（Constraints）
- 结构（Structure）
- 验证（Verification）

## 1) 三段式：Role / Task / Output
一个稳定的提示通常包含：

- 角色：你是谁、权限边界是什么
- 任务：要解决的问题、输入字段
- 输出：严格格式（最好是 JSON Schema）

## 2) 约束优先于示例
示例会提升风格一致性，但也会带来“抄模板”。

工程建议：

- 先把输出协议写清楚
- 再用 1~2 个少量示例微调行为

## 3) 工具调用提示
当你要让模型调用工具：

- 明确什么情况下要调用
- 工具参数必须来自输入或检索，不允许编造

## 4) 与 RAG 的关系
RAG 不是“把文档贴进去”，而是“把证据贴进去”。

提示词里必须要求：

- 引用证据
- 无证据则拒答或给出澄清问题

## 自检清单
- 你的提示词有没有把输出写成机器可校验的格式？
- 你有没有写明“不得编造”的行为边界？
