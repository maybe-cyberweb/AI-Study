# Speculative Decoding：用“小模型草稿”提速推理

## 一句话理解
让一个更快、更小的“草稿模型”先写出候选 token，再由大模型快速验证与修正，从而降低 **解码阶段（decoding）** 的总体成本与延迟。

## 为什么它有效
- LLM 推理的时间主要花在 decoding：每生成 1 个 token 都要做一次前向计算（还要读 KV Cache）。
- 如果草稿模型能在“可接受的命中率”下给出正确 token，大模型就能一次性 **确认多个 token**，把多次解码合并成更少的验证步骤。

## 关键流程（概念版）
1. 草稿模型一次生成 `k` 个候选 token（draft）。
2. 大模型对这段 draft 做“批量验证”（verify）。
3. 验证通过的 token 直接接受；遇到第一个不一致 token，则回退并由大模型生成正确 token。
4. 重复循环直到完成。

## 什么时候收益最大
- **长输出**：token 数越多，潜在收益越大。
- **草稿模型足够“像”大模型**：领域一致、风格一致，命中率更高。
- **吞吐优先**：在高并发场景下可与连续批处理叠加，提升整体 QPS。

## 什么时候可能“没收益甚至变慢”
- 输出很短：验证开销抵不过收益。
- 草稿命中率过低：频繁回退，等于做了两次推理。
- 系统瓶颈不在模型：比如网络、序列化、日志 I/O。

## 工程落地要点（实战）
- **指标先行**：记录每次请求的 `draft_accept_rate`（接受率）、端到端延迟、GPU 利用率。
- **与 KV Cache 配合**：speculative 不是替代 KV Cache，而是减少“需要更新 KV 的解码步数”。
- **回退策略**：当接受率持续偏低时，自动降级到普通解码，避免浪费预算。
