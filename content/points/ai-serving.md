# LLM Serving：连续批处理、Paged KV 与吞吐优化

## 目标
把“单个请求能跑”升级成“**高并发可控**、**成本可算**、**延迟稳定**”的推理服务。

## 1) 推理服务的两段式工作负载
- **Prefill**：把提示词一次性喂给模型，构建 KV Cache，计算量大但只做一次。
- **Decoding**：逐 token 生成，循环次数多，成本由输出长度主导。

你们做干部伴读时，prefill 取决于“上下文长度”；decoding 取决于“输出长度”，二者都可能很长。

## 2) 连续批处理（Continuous Batching）
传统 batching：等齐一批再跑，延迟波动大。  
连续 batching：在每一轮 decoding 中 **动态插入新请求**，持续填满 GPU。

工程要点：
- 需要一个调度器：合并请求、管理每个请求的状态、控制每轮生成 token 数。
- 对“首字延迟 TTFT”敏感的场景，必须给新请求一定优先级，否则会被长对话“饿死”。

## 3) Paged KV / 显存管理
长上下文会让 KV Cache 占用成为第一显存大头。Paged KV 的核心思想：
- 把 KV Cache 变成可分页的块（page/block）。
- 让不同请求的 KV 在显存中更紧凑地摆放，减少碎片与浪费。

## 4) 常见吞吐瓶颈与对策
- **上下文过长**：做 Select/Compress（RAG + 摘要），而不是盲目塞历史。
- **输出过长**：用结构化输出 + 长度上限 + 分段回答，避免“写作文”。
- **并发过高**：限流 + 队列 + 降级（小模型/缓存/模板）。

## 5) 你们项目的落地建议（干部端）
- 伴读：强制“引用 + 要点 + 行动建议”的结构化输出，控制输出长度。
- 推课：把“选课/排课”分成工具链：检索 → 打分 → 校验 → 输出；减少模型长文本推理。
- 测评：以确定性计算为主（规则/统计/代码），模型只做解释与总结，降低推理成本。
