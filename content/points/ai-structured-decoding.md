# 结构化生成：Constrained Decoding / Grammar / JSON Schema

## 为什么“结构化输出”是生产系统的硬门槛
你们做推课、测评、调度时，输出不是一段好看的文字，而是：
- 一张课表（课程 ID、时间段、原因）
- 一份测评结果（维度、分数、证据）
- 一条调度指令（资源、优先级、冲突解决）

一旦格式不稳定，就会出现“静默失败”“幻觉级联”“不可追溯”。

## 1) 三层约束：从弱到强
1. **提示词约束**：告诉模型“请输出 JSON”。弱，模型仍可能跑偏。
2. **后处理校验**：用 JSON Schema/Pydantic 校验，不合格就重试。中等，能把错误拦住，但会消耗重试预算。
3. **解码阶段约束（Constrained Decoding / Grammar）**：在 token 生成阶段就限制只能走合法路径。强，格式稳定性最高。

## 2) 典型做法：JSON Schema 强校验 + 有限重试
在 Harness 里做确定性校验：
- 字段类型（string/number/array）
- 枚举值（课程等级、岗位类型）
- 业务约束（课程 ID 是否存在、时间段是否冲突）

错误要“可读”地返回给模型，让它能修正：
- 哪个字段错了
- 期待的格式是什么
- 给一个最小示例

## 3) 结构化输出的“陷阱”
- 模型输出 JSON 里夹杂解释文字：需要严格提取 JSON 区块或直接强制只输出 JSON。
- 数值单位不一致：必须在 schema 中明确（例如分数 0-100，时长分钟）。
- 日期时间：统一格式（ISO 8601）并校验时区。

## 4) 用在你们五大模块里
- **推课**：输出 `recommended_courses[]`，每项含 `course_id/reason/prerequisites`。
- **测评**：输出 `dimensions[]`，每项含 `name/score/evidence_ids`。
- **档案**：输出 `learning_event`，并写入不可篡改审计日志。

## 自检清单
- 我能写出一份 JSON Schema 描述“推课结果”吗？
- 我能在后端做校验与熔断，并把错误反馈给模型重试吗？
- 我能限制重试次数与 token 预算，避免死循环烧钱吗？
